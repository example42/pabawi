Commit Message

refactor(backend): centralise secrets in ConfigService

Migrate all route handlers from direct process.env.JWT_SECRET and process.env.PABAWI_LIFECYCLE_TOKEN reads to ConfigService accessors (getJwtSecret(), getLifecycleToken()). Add Zod-validated schema fields for both secrets. Refactor AuthenticationService to remove its internal process.env fallback — the secret is now always provided by the caller via ConfigService. Update DIContainer.createDefaultContainer() to register ConfigService. Fix lint errors (unnecessary type assertions in server.ts, journal.ts, DIContainer.ts). Update vitest env and test helpers to supply required secrets.

Impact Analysis

Blast radius: medium-high auth infrastructure

This change touches the authentication secret plumbing across all authenticated route handlers. The change is structurally cross-cutting (8 route files + AuthenticationService + server.ts) but behaviourally narrow: the runtime value of the JWT secret is unchanged — only its source moves from process.env to ConfigService. The lifecycle token auth in inventory.ts is refactored from a module-level function to a factory, which is a slightly deeper structural change but preserves identical request/response semantics.

Risk: If ConfigService fails to load (missing env vars), the app will fail at startup with a Zod validation error rather than silently running with undefined secrets. This is intentional — fail-fast is safer than silent degradation for auth secrets.
