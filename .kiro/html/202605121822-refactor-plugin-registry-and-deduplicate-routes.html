refactor(backend): plugin registry loop & deduplicate routes
Replace nine copy-pasted plugin initialisation blocks in server.ts with a declarative pluginRegistry array and a single for...of loop. This reduces server.ts by ~660 lines and makes adding new integrations a one-entry change. Remove duplicate router mounts: - /api/nodes inventory mount (keep /api/inventory) - /api/nodes packages mount (keep /api/packages) Update all frontend consumers to use canonical API paths: - /api/nodes/:id → /api/inventory/:id - /api/nodes/:id/lifecycle-actions → /api/inventory/:id/lifecycle-actions - /api/package-tasks → /api/packages/package-tasks - /api/nodes/:id/install-package → /api/packages/:id/install-package Update affected tests to match new paths. Implements spec tasks 4.1, 4.2, 6.1.

Impact Analysis

Blast Radius

  • server.ts — major structural change but behaviour-preserving. All plugin init logic moved to registry.ts; the loop produces identical IntegrationManager state.
  • Route deduplication — /api/nodes mounts removed. Any external client or bookmark using /api/nodes/:id will get 404. Frontend paths updated to /api/inventory.
  • Plugin retrieval — downstream code now uses integrationManager.getExecutionTool() / getInformationSource() instead of local variables. Type-safe but changes the lookup path.

Risk Assessment

  • MEDIUM Route path changes — breaking change for any external API consumers using /api/nodes
