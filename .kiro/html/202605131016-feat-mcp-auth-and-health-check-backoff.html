What to Check

  • MCP auth with static token: Set MCP_AUTH_TOKEN=test123 in .env, restart backend, verify curl -X POST http://localhost:3000/mcp -H "Authorization: Bearer test123" returns MCP protocol response (not 401)
  • MCP auth without token: Remove MCP_AUTH_TOKEN, restart, verify JWT auth still works on /mcp
  • MCP auth rejection: Send request with wrong bearer token → expect 401
  • Health check adaptive scheduling: With an unhealthy integration (e.g., bad PuppetDB URL), verify logs show retry interval (60s) being used. With all healthy, verify 5-minute interval.
  • Log noise reduction: Verify AWS/Azure/Proxmox credential validation no longer logs at INFO during health checks (only at DEBUG)
  • Backend startup: Verify clean startup with MCP_ENABLED=true and both with/without MCP_AUTH_TOKEN

Impact Analysis

MCP Authentication (new middleware)

Isolated to /mcp routes only. The middleware is mounted exclusively on POST/GET/DELETE /mcp — no other routes are affected. Falls through to existing JWT auth when static token is not configured or doesn't match, so existing JWT-based MCP usage continues to work.

Health Check Scheduler (IntegrationManager)

Changed from setInterval (fixed cadence) to recursive setTimeout (adaptive). Blast radius is limited to IntegrationManager class — no callers change. The scheduler is started once in server.ts after plugin init. Behavioral change: unhealthy plugins now get rechecked every 60s instead of every 5min, reducing time-to-recovery detection.

Log level downgrades (AWS/Azure/Proxmox)

Purely cosmetic — reduces log noise during periodic health checks. No functional impact.

Config schema addition

mcpAuthToken is optional with no default. Existing deployments without the env var are unaffected — the field resolves to undefined and the middleware falls through to JWT.
