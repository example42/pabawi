Commit Message

feat(mcp): add facts_bulk tool and bulk facts endpoint
Add a new `facts_bulk` MCP tool and corresponding REST endpoint (`GET /api/integrations/puppetdb/facts/bulk`) that retrieves specific facts across all nodes in a single PuppetDB query. This eliminates N+1 per-node requests that triggered 429 rate limiting. Frontend changes: - Refactor GlobalFactsTab to use the bulk endpoint instead of per-node fact fetching, reducing requests from N to 1 - Simplify loading/error state from per-node maps to single booleans Backend fixes: - Add eslint-disable comments for legitimate runtime guards (window.localStorage checks in non-standard environments) - Narrow error types in Promise.reject and Object.entries calls - Remove redundant String() wraps and nullish coalescing on values already guaranteed by types - Add eslint-disable for non-null assertions in MCP transport callbacks where the SDK guarantees the value exists

Impact Analysis

New surface area: PuppetDBService.getBulkFacts() is called by both the MCP tool handler and the REST route. If PuppetDB is unavailable, both paths return appropriate errors.

Blast radius of type fixes: Low. The changes are narrowing types (removing any inference, adding explicit generics) and adding eslint-disable comments for known-safe patterns. No runtime behavior changes.

GlobalFactsTab refactor: Medium. The component's data flow changed from per-node Maps to a single bulk object. The loading UX is simpler (one spinner vs per-cell spinners). If the bulk endpoint fails, the entire table shows an error rather than individual cell errors.

Integrations touched: PuppetDB (new method + route), Proxmox (type fix), AWS (type fix), Azure (type fix), Hiera (type fix), Puppetserver (type fix). Only PuppetDB has functional changes; the rest are type-safety improvements.
